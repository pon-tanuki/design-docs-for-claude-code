# プロジェクト固有設定

## 言語設定
- すべてのドキュメントは日本語で記述してください

## ドキュメント編集ルール

### ドキュメント情報の更新
ドキュメントを編集する際は、必ず以下を更新してください：
- **最終更新日**: 編集日を`YYYY-MM-DD`形式で記入
- **変更履歴**: ドキュメント末尾の変更履歴テーブルに変更内容を追記

### ドキュメント構造
- 見出しは階層的に整理（H1は1回のみ、H2→H3→H4の順）
- 目次は必ず含める
- 関連ドキュメントへのリンクを適切に設定

### コードブロック
- 必ず言語指定を行う（\`\`\`typescript, \`\`\`python など）
- 実際に動作する実践的なコード例を提供
- コメントは日本語で記述

### Mermaid図
- 複雑な概念はMermaid図で視覚化
- 図には適切な日本語ラベルを付ける
- 利用可能な図：
  - フローチャート（flowchart）
  - シーケンス図（sequenceDiagram）
  - ER図（erDiagram）
  - クラス図（classDiagram）

### 例
\`\`\`mermaid
graph TB
    A[要件定義] --> B[設計]
    B --> C[開発]
    C --> D[テスト]
    D --> E[リリース]
\`\`\`

## バージョン管理
- セマンティックバージョニングを使用（major.minor.patch）
- 破壊的変更はメジャーバージョンアップ
- 機能追加はマイナーバージョンアップ
- バグ修正はパッチバージョンアップ

## Git コミットルール
- Conventional Commits形式を使用
  - `feat:` 新機能
  - `fix:` バグ修正
  - `docs:` ドキュメント変更
  - `refactor:` リファクタリング
  - `test:` テスト追加・修正
- コミットメッセージは日本語で詳細に記述
- Co-Authored-Byを含めない

## ドキュメントフェーズ

### 01_planning（計画）
- プロジェクト概要書
- 要件定義書
- ユースケース記述書

### 02_design（設計）
- システム設計書
- データベース設計書
- API仕様書
- 画面設計書

### 03_development（開発）
- コーディング規約
- 開発環境構築手順
- 技術仕様書

### 04_testing（テスト）
- テスト計画書
- テストケース仕様書
- テスト報告書

### 05_operation（運用）
- 運用手順書
- 障害対応手順書
- 保守・メンテナンス計画書

### 06_common（共通）
- README
- CHANGELOG
- ADR（アーキテクチャ決定記録）
- セキュリティ設計書
- パフォーマンス設計書

## 禁止事項
- 機密情報（.env、パスワード、APIキー等）の読み書き
- .gitディレクトリの直接操作
- 破壊的なBashコマンド（rm -rf、git reset --hard等）
- 強制プッシュ（git push --force）

## カスタムコマンド使用方法

### /update-doc <ファイル名> <変更内容>
ドキュメントを更新し、自動的に最終更新日と変更履歴を更新します。

例: `/update-doc docs/01_planning/project_overview.md 予算を100万円に変更`

### /check-doc <ファイル名>
ドキュメントの品質をチェックし、問題があれば修正提案を行います。

例: `/check-doc docs/02_design/system_design.md`

### /new-phase-doc <フェーズ> <ドキュメント種類>
新しいフェーズのドキュメントをテンプレートから作成します。

例: `/new-phase-doc planning project_overview`
